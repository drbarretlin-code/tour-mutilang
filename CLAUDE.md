# CLAUDE.md

本檔案除作為 Claude Code 的專案說明文件外，亦為 App 行程規劃業務規則的**唯一來源（single source of truth）**。下方「旅遊行程規劃優先級與流程規範」為規則式行程引擎（[src/services/ai.ts](src/services/ai.ts)）設計與調整時所應遵循的規範依據。

## Claude Code 協作規範

- 與使用者對話時，請一律使用繁體中文回覆。

---

## 目標平台規範

- 本應用程式未來上架目標以 **Windows 與 macOS 兩大桌面平台**為主，並須**同步滿足 iOS 與 Android** 平台。
- Web 版本僅作為**開發階段檢視（preview/debug）用途**，並非正式上架目標平台。
- 因此，所有開發流程、相依套件選擇與程式邏輯設計，皆須以**同時滿足 Windows、macOS、iOS、Android 四大平台**為優先考量；Web 相容性僅為開發便利性的附帶效果，不可作為功能設計或取捨的主要依據。

---

## 響應式設計規範 (Responsive Design)

- 本應用程式之所有畫面與元件**必須遵循響應式設計原則**，確保在桌面（Windows/macOS，寬視窗）、平板與手機（iOS/Android，窄視窗）等不同尺寸與方向下，皆能正確排版、可讀且可操作。
- **實作基準**：版面斷點與裝置尺寸判斷一律透過共用的 [useResponsive](src/hooks/useResponsive.ts) hook（如 `isLargeScreen`）取得，**不得**在各元件硬編魔術數字或自行重複判斷；新增畫面時應沿用既有的斷點語意。
- **佈局原則**：採彈性佈局（flex / 百分比 / `flexWrap`）與相對單位，避免固定寬高造成溢出或裁切；多欄版面在窄螢幕應自動降級為單欄堆疊（參考 [DestinationGuide](src/components/itinerary/DestinationGuide.tsx) 之 `Platform.OS === 'web' ? 'row' : 'column'` 模式，但應優先以螢幕寬度而非平台判斷）。
- **觸控與可及性**：可點擊元素須具備足夠的觸控目標尺寸，文字需可隨系統字級縮放，避免因縮放而破版。
- 任何新增或調整 UI 的提交，皆須在寬／窄兩種視窗下檢視無破版後方可視為完成。

# 旅遊行程規劃優先級與流程規範

為確保行程產出與編修時具備邏輯一致性，系統與 AI 必須嚴格遵守以下優先順序與流程規範：

## 一、 行程節點優先順序 (Node Priority)

當行程規劃中不同節點的時間、位置或邏輯發生衝突時，必須遵循以下優先級進行妥適排程，低優先級之節點應配合高優先級之節點進行時間與動線的調整：

1.  **第一優先：航班資訊（去程與返程航班）**
    *   第一天的去程航班抵達時間為整個行程的起點，後續所有活動必須在完成通關（預估 90 分鐘）後方能開始。
    *   最後一天的返程航班出發時間為整個行程的終點，旅客必須於起飛前 150 分鐘抵達機場，當天所有前置活動必須在此時間前結束。
2.  **第二優先：特定地點需求**
    *   使用者於問卷中填寫之「特定地點需求」（包括指定飯店、特定日期與時間、指定停留時間與距離備註），具有極高權重，優先順序大於必訪景點。
    *   若在此處指定了預訂的飯店，AI 規劃時應以此飯店作為每日行程起訖之唯一基準點（滿足每日起訖點規範）。
3.  **第三優先：必訪景點**
    *   使用者於計畫中勾選之必訪景點必須 100% 納入行程規劃。
    *   必訪景點的排程順序與時間應優先於上傳檔案與參考網址。
4.  **第四優先：上傳檔案內容**
    *   由使用者上傳之行程文件、預訂確認單或相關檔案所擷取之景點與活動。
5.  **第五優先：參考網址**
    *   使用者提供之參考網頁或外部連結所提及之景點與活動。

## 二、 每日起訖點規範 (Hotel & Transportation Loop)

為確保旅客住宿與行李移動的便利性，行程的每日起訖點必須符合以下迴圈規範：

1.  **首日與尾日交通銜接**：
    *   **第一天（首日）**：行程必須由機場（或港口、火車站等交通樞紐點）開始，結束於住宿點。
    *   **最後一天（尾日）**：行程必須由住宿點開始，並以機場（或同首日之交通樞紐點）作為終點結束。
2.  **首日抵達後須先辦理入住**：
    *   第一天完成機場通關後，下一個行程節點**必須是前往當晚住宿地點辦理入住（寄放行李）**，待安頓妥當後才展開當日其餘景點行程。此安排符合「先安頓住宿、再展開行程」之現實旅遊邏輯，避免旅客拖著行李四處移動。
3.  **中間天數住宿閉環**：
    *   除首日與尾日外，所有中間天數的行程安排，均必須由當日之**住宿點開始**，並於當日行程結束後**回到住宿點結束**。
4.  **住宿區間之入住與退房邏輯**：
    *   使用者指定之住宿若包含日期區間（例如 `2024-01-01 ~ 2024-01-03`），該區間之**首日為入住日、末日為退房日**，退房當日不視為續住。
    *   退房當日之行程**起點**仍為該住宿點（早上由此出發），但當日行程結束後之**回程終點（晚上入住）應改為下一段住宿區間所指定之住宿點**；若使用者未指定下一段住宿，則退回 AI 建議之預設住宿點。
    *   若退房日恰為行程最後一天，則依「首日與尾日交通銜接」規範，當日終點為機場（或交通樞紐點），不適用上述回程終點邏輯。
5.  **住宿點優化建議**：
    *   AI 必須主動依據使用者於「我的計劃」中所選定之「總預算」範圍（例如：經濟、奢華等），在規劃行程時，主動提出最適化之建議住宿點，並將該住宿點作為當日行程的起訖依據。

## 三、 行程時間安排限制 (Time Constraints)

1.  **每日活動區間**：每天行程的活動時間安排，必須嚴格限制於當地時間的 **08:00 至 21:00** 之間（不含夜間睡眠時間）。
2.  **合理間隔**：景點與景點之間必須預留合理的交通轉乘與休息時間，避免行程過度緊湊。
3.  **回到住宿點時間之推算**：「回到住宿點」此一節點的時間，**必須依據當日最後一個行程活動之結束時間，加上合理的交通轉乘時間順延推算**，不可採用與當日實際行程脫鉤的固定時間；最終仍須符合上述每日 08:00-21:00 之區間限制，超出範圍時應整體校正而非單獨裁切該節點。

## 四、 多國語系相容規範 (Localization Rules)

1.  **文字顯示對齊**：所有新增、編輯或由 AI 產生的行程內容（包含景點名稱、景點描述、交通說明與注意事項），均必須符合多國語系顯示之要求，並對齊使用者在問卷中所設定之語系（Locale）。
2.  **官方本地名稱**：在非母語地區旅遊時，景點的 `localTitle` 欄位必須精確提供目的地官方使用的本地語言名稱，以供旅客於現場叫車或地圖搜尋時使用。

## 五、 機場導覽與租/包車/叫車平台匹配規範 (Airport Guide & Ride-Hailing Rules)

1.  **動態機場指引**：系統必須根據當日地區（`region`）、活動地點名稱與活動標題，動態載入並顯示正確的機場指引。系統已內建全球主要機場（含日本成田 NRT、羽田 HND、關西 KIX；台灣桃園 TPE、松山 TSA；泰國蘇凡納布 BKK、廊曼 DMK；韓國仁川 ICN、金浦 GMP；新加坡樟宜 SIN；英國倫敦希斯洛 LHR；美國紐約甘迺迪 JFK；法國巴黎戴高樂 CDG 等）的精緻指引，並具備 Generic Fallback 解析器：自動從地名中提取 3 字機場代碼、根據關鍵字對齊國旗，並生成通用的雙語入境與離境指引，徹底避免硬編碼。
2.  **租/包車/叫車平台動態匹配**：不同國家之交通推薦應動態顯示該國主流叫車 App 名稱與連結，並支援對應的 App Deep Link（網頁版則降級提示）：
    *   **日本**：推薦 **GO** (App scheme: `taxigo://`) 與 **Uber**。
    *   **台灣**：推薦 **yoxi** (App scheme: `yoxi://`) 與 **Uber**。
    *   **泰國**：推薦 **Grab** (App scheme: `grab://`) 與 **Bolt** (App scheme: `bolt://`)。
    *   **其他地區**：預設為 **Uber** 與 **Google 地圖叫車**。

## 六、 OTM 景點資料與本地模板翻譯對齊規範 (OTM POI Translation Alignment)

1.  **線上抓取翻譯補全**：當系統透過 OpenTripMap (OTM) 線上 API 取得英文 POI 資料後，為符合 UI 語系顯示名稱之要求，必須將其與本地離線資料庫（[destinations.json](src/data/destinations.json)）比對（比對規則優先以座標經緯度差值 < 0.005 度作為精準匹配，次要以各語系文字模糊匹配）。
2.  **附加 UI 顯示名稱**：匹配成功時，必須將 API 取得的標題替換或附加為 UI 語系當地名稱，並將目的地官方使用的本地名稱作為 `localTitle` 保留，確保旅客可於現場叫車或搜尋地圖使用。
3.  **無匹配景點之動態美化**：若離線資料庫查無匹配，系統應調用規則式美化翻譯器（Blogger Style Enhancer），解析英文詞根字尾（如 `Museum` 轉化為符合該語系習慣之 `博物館 (知性文藝之旅)`、`Palace` 轉化為 `古典宮殿 (皇家歷史漫步)`）或熱門地標，為其組裝成符合該 UI 語系之部落客風格「接地氣」名稱，取代乾癟音譯，並將原名保留於 `localTitle` 作為附加顯示。

## 七、 景點描述個性化規範 (POI Description Enrichment)

1.  **優先使用內建獨特描述**：當 OTM POI 經座標或名稱比對命中本地離線資料庫（`destinations.json`）中的景點或餐廳時，必須優先使用該筆資料的 `desc` 欄位所提供的獨特景點描述（每個景點量身撰寫、300 字以上），取代依分類產生的通用描述。此規則確保每個景點的介紹內容具備差異化，避免同類型景點（如寺廟、公園、商圈）的說明千篇一律。
2.  **分類式描述為最後退路**：僅當離線資料庫查無匹配時，才退回 `buildPoiDescription` 依分類（religion/historic/museum/nature 等）組裝的通用描述。
3.  **維基百科摘要擇長覆寫**：無論使用內建獨特描述或分類式描述，後續的維基百科摘要補強流程（`fetchWikipediaSummaries`）仍會執行，且僅在維基摘要長度≥既有描述時才進行覆寫，確保描述品質只增不減。
4.  **描述資料來源優先序**：`destinations.json 內建獨特描述 > buildPoiDescription 分類式描述 > Wikipedia 摘要（擇長覆寫）`。

## 八、 行程興趣覆蓋與動態替換規範 (Interest Coverage & Dynamic Re-roll Rules)

1.  **總體行程興趣全覆蓋**：在整個計劃中（所有天數總和），景點安排必須**完全滿足**使用者在「興趣與活動」市場所勾選的全部選項。每一個被勾選的選項，至少要安排一處景點以上。系統應依照實際的路線、區域以及旅客可能的體力負荷，來決定每天的景點安排種類與適當分配。
2.  **多元周邊「換一下」推薦**：當使用者點擊「換一下」時，系統提供的替換景點選項**不應侷限於原本景點的單一性質**。替換清單必須滿足以下條件：
    *   **空間範圍**：以當下被替換的景點位置為中心，外擴 1 公里範圍。
    *   **推薦依據**：以使用者最初在「興趣與活動」內勾選的**全部**選項內容為基準進行搜尋。
    *   **數量與品質**：盡可能為每一個勾選的選項擇優推薦一個地點，且所有「換一下」的候選建議清單，總數**必須至少達到 8 個以上**，確保使用者有充足且符合跨興趣的周邊選項。
